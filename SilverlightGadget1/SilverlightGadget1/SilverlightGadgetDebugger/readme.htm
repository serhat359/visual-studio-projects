Introduction

The purpose of the SilverlightGadgetDebugger project is only to act as a debug session starter for the Silverlight gadget. Even though it is a console application project, it produces no output. Instead, whenever the project is started it will invoke the SidebarSandboxStarter.exe file. This application receives the current solution path as a command line parameter. It uses this information to connect to the Visual Studio that has the specified solution opened. The SidebarSandboxStarter will attach the debugger from that instance of Visual Studio to the Windows Sidebar process (sidebar.exe). If the process is not started, it will start it.

NOTE: Silverlight gadgets that use a Sivlerlight version older than 5.0 only work in the 32 bit version of the Sidebar. Template will attach to the 32 bit or 64 bit version of the Sidebar, depending on the value of Platform Target setting of the SilverlightGadgetDebugger project (from the property pages of the project).

  • AnyCPU : 32 bit Sidebar
  • x86 : 32 bit Sidebar
  • x64 : 64 bit Sidebar (only on 64 bit OS, otherwise, 32 bit)
  • Itanium: 64 bit Sidebar (only on 64 bit OS, otherwise, 32 bit)

Adding More Content to the SilverlightGadgetWeb Project

  • Include the content in the project
  • Set the "Build Action" to "Content"
  • Set the "Copy to Output Directory" to "Copy always" or "Copy if newer"

How to Debug Silverlight Gadgets Built Using This Template

  • Make sure that the SilverlightGadgetDebugger is set as the startup project in the solution.
  • Make sure that the proper Platform Target is set for the SilverlightGadgetDebugger project. For Silverlight versions older than 5.0, only AnyCPU or x86 will work.
  • Start debugging this project (F5) and eventually Visual Studio will automatically attach to the Sidebar. At this point you can debug.
  • When you stop debugging the project, the Sidebar is restored to the previous state.
  • When you run or debug using the method described in this file, you will get a UI with a "Close" button that you can use to terminate the run or debug session.

Sometimes it can take a bit longer (though not more than aprox. 20 seconds) for Visual Studio to attach to the Sidebar. If you feel that the debugger is taking too long to attach, kill the SidebarSandboxStarter.exe process and try again.

If you start the SilverlightGadgetDebugger project without debugging (Ctrl+ F5), it is possible that you will get a console window that will eventually prompt you to "Press any key to continue...". You can close this window and there won't be any side-effects.

As an alternative, you can manually attach to the 32 bit Sidebar process (without involving the SilverlightGadgetDebugger project at all).

How to Publish Silverlight Gadgets Built Using This Template

Each time the SilverlightGadgetWeb project is built, a corresponding .gadget deployment file is generated. The file is located under SilverlightGadgetWeb\bin\Publish.

Troubleshooting

Please don't move or change the name of the SilverlightGadgetDebugger project, as the build events from the template rely on the project's name (and relative path within the solution).

If you make changes to some of the template generated files that break the build of the .gadget file or the F5 debugging experience, you can create a new project using the template and compare your changes with the original working versions.

If you change the Platform Target, unload/reload or close/open the SilverlightGadgetDebugger project. Otherwise, the change will not be picked up by the starting arguments of the project (limitation in Visual Studio).
